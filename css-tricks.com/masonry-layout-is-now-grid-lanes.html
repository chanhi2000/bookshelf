Masonry Layout is Now grid-lanes
It's settled! A new CSS display property keyword called grid-lanes will trigger a masonry layout mode.

We‚Äôve talked a lot about Masonry layout here on CSS-Tricks, so there‚Äôs no need to remind you of the two-sided discussions about how to approach it, the idea of using it as a new unifying layout concept, or alternative approaches for making it work today.

Here‚Äôs what you do need to know: it‚Äôs going to be display: grid-lanes.

The earliest talks concerning masonry began in 2017 when Rachel Andrew expressed some concerns about how to make a Pinterest masonry layout (w3c/csswg-drafts). Rachel said it felt like the right approach should be a flexbox kind of thing, but this wasn‚Äôt fully achievable with flexbox, as items would rather flow from top-to-bottom rather than across each row:

Currently the closest you can get with CSS to this type of layout is to use multi-col however, the items then flow top to bottom rather than across the rows.

This feels more like a behaviour of flexbox than grid, as the solution is very much based on the size of the items. Opening this in order to record the feature request/use case for future discussion.

And so, the talks surrounding masonry layout in CSS began within the W3C. In that same GitHub thread, you‚Äôll read that Tab Atkins-Bittner (w3c/csswg-drafts) shared a ‚Äúhacky‚Äù way to go about it with CSS Grid.

You can have Grid do Masonry if you know the heights of the items ahead of time. It‚Äôs a bit hacky, but you set the grid container to grid: auto-flow dense 1px / <column widths here>;, then set each item to grid-row: span <pixel height as integer>; (for example, if the item is 50px tall, use grid-row: span 50;).

I‚Äôm a sucker for clever CSS hacks, but you know what else I‚Äôm a sucker for? A working CSS solution that makes it easier for beginners to use. And no, unfortunately, we cannot use grid-template-rows or grid-template-columns, as Nate Green suggested (w3c/csswg-drafts) in the same thread:

.figure-list {
  display: grid;
  grid-template-columns: 1fr 1fr 1fr;
  grid-template-rows: flow;
}

Atkins-Bittner replied (github.com/w3c/csswg-drafts)

There‚Äôs no ‚Äúsimple‚Äù way to adapt Grid into Masonry ‚Äì anything would involve non-trivial edits to the layout algorithm. Packery, in particular, really needs things to have a defined width.

Okay, that‚Äôs fine. I‚Äôm sure you‚Äôd be thinking as well, ‚Äúwhy not create a new display property for it?‚Äù Well, Rachel added (w3c/csswg-drafts) that:

The thing is we can‚Äôt create a brand new value of display for every distinct design pattern.

Mind you, all these discussions were happening as far back as 2017, meaning masonry has been an elusive goal for quite some time.

Just to make sure you‚Äôre with me, this is an example image of the kind of layout masonry looks like (thanks to Michael Richins for this (w3c/csswg-drafts)):

One thing I would like is that prior to the announcement of masonry display type, there were solutions/hacks discussed, like this one from Andy Barefoot (w3c/csswg-drafts), but nothing truly was decided as the main go-to feature for CSS masonry. Hey, there was even a super cool Masonry library from David DeSandro (desandro), and he gave pretty useful advice to W3C (w3c/csswg-drafts) and browser vendors when writing the spec and implementing the masonry layout, like on how loading images would work, multi-column spanning items, filling gaps, retaining horizontal order.

This thread helped, but it still didn‚Äôt lead to a conclusive statement about what the masonry syntax will look like. In fact, we had code suggestions, like from Dan Tonon (w3c/csswg-drafts):

.flex-container {
  display: flex;
  flex-direction: column;
  flex-block-count: 2; 
  flex-block-flow: cross; 
}

This introduces two new flexbox properties called flex-block-count and flex-block-flow which allows us to control items similarly (with column-count) and control the flow of items, respectively. However, this wouldn‚Äôt fly as Michael Richins pointed out (w3c/csswg-drafts):

This will still not work for children of different height‚Ä¶

This thread was not going anywhere, so discussions ended in April 2020 with a GitHub user with the name Nic787 referencing and agreeing with an earlier (w3c/csswg-drafts) probable (w3c/csswg-drafts) solution (w3c/csswg-drafts) for masonry using float instead of grid.

I think you have a good point. Currently, float is becoming old and flexbox allow a lot of things to be done, but this masonry layout is lacking in both.

‚Ä¶ Sometimes you can have lot of small images in a row, so making it left to right can‚Äôt work all the time. Masonry is left to right like always, but in a way it‚Äôs easier for the user to access informations.

Fortunately, discussions still continued in many threads (w3c/csswg-drafts), and the requests for the masonry display property flooded W3C. Another notable GitHub thread that surfaced on the masonry debate is, well, the Masonry Syntax Debate (w3c/csswg-drafts) in November 2024, but that also didn‚Äôt lead to much at that point (w3c/csswg-drafts). In fact, Jen Simmons noted in the meeting (w3c/csswg-drafts) that:

Personally disappointed that we‚Äôre not making more progress. We‚Äôve been having this argument for 5 years.

Five years of back and forth on how exactly the naming and structure would be. Do we use grid? Do we use a brand new property? Do we create separate properties just for masonry? We couldn‚Äôt really decide on that.

One thing I like to get behind is a comment from our very own Juan Diego, stating how [solutions aren‚Äôt ‚Äújust use grid‚Äù or create a new property (`w3c/csswg-drafts)](https://github.com/w3c/csswg-drafts/issues/11243#issuecomment-2524100674), but that we could get rid of the `grid-` prefix altogether and settle for an option that caters for both `grid` and `masonry`:

‚Ä¶it should be possible to use a new unprefixed template-areas property for masonry and grid, regardless of the formatting context..

Fortunately, something good came out of that thread because another meeting (w3c/csswg-drafts) was held on January 31st of 2025, and guess what? It was resolved to reuse grid templating and placement properties for masonry layout.

The CSS Working Group just discussed [css-grid-3][masonry] Masonry Syntax Debate, and agreed to the following: RESOLVED: Re-use grid templating and placement properties for masonry layout.

Wait. Why am I announcing about masonry syntax properties? What about the main syntax itself? Isn‚Äôt that why we‚Äôre here? Awesome, you‚Äôre still with me. So let me show you the exact thread where the masonry syntax war ended.

It‚Äôs (almost) here!

This brings us to the latest announcement (w3c/csswg-drafts) that setting grid-lanes on the display property activates a masonry layout.

.masonry {
  display: grid-lanes;
}

It was tough to get here. Deciding on the keyword alone took years. Just look at all of the names (w3c/csswg-drafts) that were considered leading up to grid-lanes:

  • collapsed-grid or grid collapse
  • grid-stack or stacked-grid
  • grid-pack or packed-grid or grid pack
  • compact-grid or compact grid
  • grid-masonry or masonry-grid
  • grid-flex or flex-grid
  • grid single-axis (masonry grid) vs. grid dual-axis (normal grid)
  • grid stack (masonry grid) vs. grid grid (normal grid)
  • staggered-grid or grid staggered
  • uneven-grid
  • semi-grid
  • lane-grid or grid-lanes
  • axial-grid

I wouldn‚Äôt have minded staggered-grid since that‚Äôs what it is in React Native. In fact, Kevin Powell agreed (w3c/csswg-drafts). That said, I am perfectly okay with grid-lanes. It will take some time for browsers to implement it, as discussions surrounding the shorthand adjustments for masonry are still going on (w3c/csswg-drafts), with nothing much said on the issue just yet.

What about Item Flow?

Oh yeah, remember that? It‚Äôs still in the works (w3c/csswg-drafts), and I expect that we‚Äôll see it after a proper grid-lanes implementation starts making the rounds. There‚Äôs a nice hypothetical example from Fantasai (w3c/csswg-drafts) illustrating how it might look if added to the spec today:

item-flow: <item-direction> || <item-wrap> || <item-pack> || <item-slack>

/* shorthand for */
item-direction: row | column | row-reverse | column-reverse
item-wrap: wrap | wrap-reverse
item-pack: normal | dense || collapse
item-slack: <length-percentage>

In use, it would look something along the lines of:

.masonry {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(14rem, 1fr));
  item-flow: collapse; /* == item-flow: row collapse 1em; */
  gap: 1rem;
}

So, what‚Äôs next?

After many back-and-forths, masonry grid-lanes is here! But where exactly?

Well, this is the awkward part. The truth is we‚Äôll still have to wait for browsers to implement grid-lanes. How long may it take? I really can‚Äôt say, but take into consideration that all major browsers already have implemented some type of masonry layout behind flags or previews, but with different syntaxes:

  • The Chrome and Edge teams initially implemented Masonry in Chromium 140 as display: masonry but are already working on switching to grid-lanes. Some of their demos (MicrosoftEdge/Demos) already reflect it.
  • The WebKit team implemented masonry in Safari 17 as display: grid, but they already seem to have some work on grid-lanes in the pipelines.
  • The Mozilla team was actually the first browser to implement masonry layout in 2020 ‚Äî using display: grid as well. But like the others, they‚Äôre already in the process of making the switch to grid-lanes.

So, good news all around! All that‚Äôs left is to wait. If you want to track grid-lanes progress, here is a good link pool for each browser from Patrick Brosset. And, if you can‚Äôt wait for the official masonry implementation, Zell Liew has an approach for using it today, with minimal JavaScript.

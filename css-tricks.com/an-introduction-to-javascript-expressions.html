Hey, I‚Äôm Mat, but ‚ÄúWilto‚Äù works too ‚Äî I‚Äôm here to teach you JavaScript.

Well, not here-here; technically, I‚Äôm over at JavaScript for Everyone to teach you JavaScript. What we have here is a lesson from the JavaScript for Everyone module on lexical grammar and analysis ‚Äî the process of parsing the characters that make up a script file and converting it into a sequence of discrete ‚Äúinput elements‚Äù (lexical tokens, line ending characters, comments, and whitespace), and how the JavaScript engine interprets those input elements.

An expression is code that, when evaluated, resolves to a value. 2 + 2 is a timeless example.

2 + 2
// result: 4

As mental models go, you could do worse than ‚Äúanywhere in a script that a value is expected you can use an expression, no matter how simple or complex that expression may be:‚Äù

function numberChecker( checkedNumber ) {
  if( typeof checkedNumber === "number" ) {
    console.log( "Yep, that's a number." );
  }
}

numberChecker( 3 );
// result: Yep, that's a number.

numberChecker( 10 + 20 );
// result: Yep, that's a number.

numberChecker( Math.floor( Math.random() * 20 ) / Math.floor( Math.random() * 10 ) );
// result: Yep, that's a number.

Granted, JavaScript doesn‚Äôt tend to leave much room for absolute statements. The exceptions are rare, but it isn‚Äôt the case absolutely, positively, one hundred percent of the time:

console.log( -2**1 );
// result: Uncaught SyntaxError: Unary operator used immediately before exponentiation expression. Parenthesis must be used to disambiguate operator precedence

Still, I‚Äôm willing to throw myself upon the sword of ‚Äúum, actually‚Äù on this one. That way of looking at the relationship between expressions and their resulting values is heart-and-soul of the language stuff, and it‚Äôll get you far.

Primary Expressions

There‚Äôs sort of a plot twist, here: while the above example reads to our human eyes as an example of a number, then an expression, then a complex expression, it turns out to be expressions all the way down. 3 is itself an expression ‚Äî a primary expression. In the same way the first rule of Tautology Club is Tautology Club‚Äôs first rule, the number literal 3 is itself an expression that resolves in a very predictable value (psst, it‚Äôs three).

console.log( 3 );
// result: 3

Alright, so maybe that one didn‚Äôt necessarily need the illustrative snippet of code, but the point is: the additive expression 2 + 2 is, in fact, the primary expression 2 plus the primary expression 2.

Granted, the ‚Äúit is what it is‚Äù nature of a primary expression is such that you won‚Äôt have much (any?) occasion to point at your display and declare ‚Äúthat is a primary expression,‚Äù but it does afford a little insight into how JavaScript ‚Äúthinks‚Äù about values: a variable is also a primary expression, and you can mentally substitute an expression for the value it results in ‚Äî in this case, the value that variable references. That‚Äôs not the only purpose of an expression (which we‚Äôll get into in a bit) but it‚Äôs a useful shorthand for understanding expressions at their most basic level.

There‚Äôs a specific kind of primary expression that you‚Äôll end up using a lot: the grouping operator. You may remember it from the math classes I just barely passed in high school:

console.log( 2 + 2 * 3 );
// result: 8 

console.log( ( 2 + 2 ) * 3 );
// result: 12

The grouping operator (singular, I know, it kills me too) is a matched pair of parentheses used to evaluate a portion of an expression as a single unit. You can use it to override the mathematical order of operations, as seen above, but that‚Äôs not likely to be your most common use case‚Äîmore often than not you‚Äôll use grouping operators to more finely control conditional logic and improve readability:

const minValue = 0;
const maxValue = 100;
const theValue = 50;

if( ( theValue > minValue ) && ( theValue < maxValue ) ) {
  // If ( the value of `theValue` is greater than that of `minValue` ) AND less than `maxValue`):
  console.log( "Within range." );
}
// result: Within range.

Personally, I make a point of almost never excusing my dear Aunt Sally. Even when I‚Äôm working with math specifically, I frequently use parentheses just for the sake of being able to scan things quickly:

console.log( 2 + ( 2 * 3 ) );
// result: 8

This use is relatively rare, but the grouping operator can also be used to remove ambiguity in situations where you might need to specify that a given syntax is intended to be interpreted as an expression. One of them is, well, right there in your developer console.

The syntax used to initialize an object ‚Äî a matched pair of curly braces ‚Äî is the same as the syntax used to group statements into a block statement. Within the global scope, a pair of curly braces will be interpreted as a block statement containing a syntax that makes no sense given that context, not an object literal. That‚Äôs why punching an object literal into your developer console will result in an error:

{ "theValue" : true }
// result: `Uncaught SyntaxError: unexpected token: ':'

It‚Äôs very unlikely you‚Äôll ever run into this specific issue in your day-to-day JavaScript work, seeing as there‚Äôs usually a clear division between contexts where an expression or a statement are expected:

{
  const theObject = { "theValue" : true };
}

You won‚Äôt often be creating an object literal without intending to do something with it, which means it will always be in the context where an expression is expected. It is the reason you‚Äôll see standalone object literals wrapped in a a grouping operator throughout this course ‚Äî a syntax that explicitly says ‚Äúexpect an expression here‚Äù:

({ "value" : true });

However, that‚Äôs not to say you‚Äôll never need a grouping operator for disambiguation purposes. Again, not to get ahead of ourselves, but an Independently-Invoked Function Expression (IIFE), an anonymous function expression used to manage scope, relies on a grouping operator to ensure the function keyword is treated as a function expression rather than a declaration:

(function(){
  // ...
})();

Expressions With Side Effects

Expressions always give us back a value, in no uncertain terms. There are also expressions with side effects ‚Äî expressions that result in a value and do something. For example, assigning a value to an identifier is an assignment expression. If you paste this snippet into your developer console, you‚Äôll notice it prints 3:

theIdentifier = 3;
// result: 3

The resulting value of the expression theIdentifier = 3 is the primary expression 3; classic expression stuff. That‚Äôs not what‚Äôs useful about this expression, though ‚Äî the useful part is that this expression makes JavaScript aware of theIdentifier and its value (in a way we probably shouldn‚Äôt, but that‚Äôs a topic for another lesson). That variable binding is an expression and it results in a value, but that‚Äôs not really why we‚Äôre using it.

Likewise, a function call is an expression; it gets evaluated and results in a value:

function theFunction() {
  return 3;
};

console.log( theFunction() + theFunction() );
// result: 6

We‚Äôll get into it more once we‚Äôre in the weeds on functions themselves, but the result of calling a function that returns an expression is ‚Äî you guessed it ‚Äî functionally identical to working with the value that results from that expression. So far as JavaScript is concerned, a call to theFunction effectively is the simple expression 3, with the side effect of executing any code contained within the function body:

function theFunction() {
  console.log( "Called." );
  return 3;
};

console.log( theFunction() + theFunction() );
/* Result:
Called.
Called.
6
*/

Here theFunction is evaluated twice, each time calling console.log then resulting in the simple expression 3 . Those resulting values are added together, and the result of that arithmetic expression is logged as 6.

Granted, a function call may not always result in an explicit value. I haven‚Äôt been including them in our interactive snippets here, but that‚Äôs the reason you‚Äôll see two things in the output when you call console.log in your developer console: the logged string and undefined.

JavaScript‚Äôs built-in console.log method doesn‚Äôt return a value. When the function is called it performs its work ‚Äî the logging itself. Then, because it doesn‚Äôt have a meaningful value to return, it results in undefined. There‚Äôs nothing to do with that value, but your developer console informs you of the result of that evaluation before discarding it.

Comma Operator

Speaking of throwing results away, this brings us to a uniquely weird syntax: the comma operator. A comma operator evaluates its left operand, discards the resulting value, then evaluates and results in the value of the right operand.

Based only on what you‚Äôve learned so far in this lesson, if your first reaction is ‚ÄúI don‚Äôt know why I‚Äôd want an expression to do that,‚Äù odds are you‚Äôre reading it right. Let‚Äôs look at it in the context of an arithmetic expression:

console.log( ( 1, 5 + 20 ) );
// result: 25

The primary expression 1 is evaluated and the resulting value is discarded, then the additive expression 5 + 20 is evaluated, and that‚Äôs resulting value. Five plus twenty, with a few extra characters thrown in for style points and a 1 cast into the void, perhaps intended to serve as a threat to the other numbers.

And hey, notice the extra pair of parentheses there? Another example of a grouping operator used for disambiguation purposes. Without it, that comma would be interpreted as separating arguments to the console.log method ‚Äî 1 and 5 + 20 ‚Äî both of which would be logged to the console:

console.log( 1, 5 + 20 );
// result: 1 25

Now, including a value in an expression in a way where it could never be used for anything would be a pretty wild choice, granted. That‚Äôs why I bring up the comma operator in the context of expressions with side effects: both sides of the , operator are evaluated, even if the immediately resulting value is discarded.

Take a look at this validateResult function, which does something fairly common, mechanically speaking; depending on the value passed to it as an argument, it executes one of two functions, and ultimately returns one of two values.

For the sake of simplicity, we‚Äôre just checking to see if the value being evaluated is strictly true ‚Äî if so, call the whenValid function and return the string value "Nice!". If not, call the whenInvalid function and return the string "Sorry, no good.":

function validateResult( theValue ) {
  function whenValid() {
    console.log( "Valid result." );
  };
  function whenInvalid() {
    console.warn( "Invalid result." );
  };

  if( theValue === true ) {
    whenValid();
    return "Nice!";
  } else {
    whenInvalid();
    return "Sorry, no good.";
  }
};

const resultMessage = validateResult( true );
// result: Valid result.

console.log( resultMessage );
// result: "Nice!"

Nothing wrong with this. The whenValid / whenInvalid functions are called when the validateResult function is called, and the resultMessage constant is initialized with the returned string value. We‚Äôre touching on a lot of future lessons here already, so don‚Äôt sweat the details too much.

Some room for optimizations, of course ‚Äî there almost always is. I‚Äôm not a fan of having multiple instances of return, which in a sufficiently large and potentially-tangled codebase can lead to increased ‚Äúwait, where is that coming from‚Äù frustrations. Let‚Äôs sort that out first:

function validateResult( theValue ) {
  function whenValid() {
    console.log( "Valid result." );
  };
  function whenInvalid() {
    console.warn( "Invalid result." );
  };

  if( theValue === true ) {
    whenValid();
  } else {
    whenInvalid();
  }
  return theValue === true ? "Nice!" : "Sorry, no good.";
};

const resultMessage = validateResult( true );
// result: Valid result.

resultMessage;
// result: "Nice!"

That‚Äôs a little better, but we‚Äôre still repeating ourselves with two separate checks for theValue. If our conditional logic were to be changed someday, it wouldn‚Äôt be ideal that we have to do it in two places.

The first ‚Äî the if/else ‚Äî exists only to call one function or the other. We now know function calls to be expressions, and what we want from those expressions are their side effects, not their resulting values (which, absent a explicit return value, would just be undefined anyway).

Because we need them evaluated and don‚Äôt care if their resulting values are discarded, we can use comma operators (and grouping operators) to sit them alongside the two simple expressions ‚Äî the strings that make up the result messaging ‚Äî that we do want values from:

function validateResult( theValue ) {
  function whenValid() {
    console.log( "Valid result." );
  };
  function whenInvalid() {
    console.warn( "Invalid result." );
  };
  return theValue === true ? ( whenValid(), "Nice!" ) : ( whenInvalid(), "Sorry, no good." );
};

const resultMessage = validateResult( true );
// result: Valid result.

resultMessage;
// result: "Nice!"

Lean and mean thanks to clever use of comma operators. Granted, there‚Äôs a case to be made that this is a little too clever, in that it could make this code a little more difficult to understand at a glance for anyone that might have to maintain this code after you (or, if you have a memory like mine, for your near-future self). The siren song of ‚ÄúI could do it with less characters‚Äù has driven more than one JavaScript developer toward the rocks of, uh, slightly more difficult maintainability. I‚Äôm in no position to talk, though. I chewed through my ropes years ago.

Between this lesson on expressions and the lesson on statements that follows it, well, that would be the whole ballgame ‚Äî the entirety of JavaScript summed up, in a manner of speaking ‚Äî were it not for a not-so-secret third thing. Did you know that most declarations are neither statement nor expression, despite seeming very much like statements?

Variable declarations performed with let or const, function declarations, class declarations ‚Äî none of these are statements:

if( true ) let theVariable;
// Result: Uncaught SyntaxError: lexical declarations can't appear in single-statement context

if is a statement that expects a statement, but what it encounters here is one of the non-statement declarations, resulting in a syntax error. Granted, you might never run into this specific example at all if you ‚Äî like me ‚Äî are the sort to always follow an if with a block statement, even if you‚Äôre only expecting a single statement.

I did say ‚Äúone of the non-statement declarations,‚Äù though. There is, in fact, a single exception to this rule ‚Äî a variable declaration using var is a statement:

if( true ) var theVariable;

That‚Äôs just a hint at the kind of weirdness you‚Äôll find buried deep in the JavaScript machinery. 5 is an expression, sure. 0.1 * 0.1 is 0.010000000000000002, yes, absolutely. Numeric values used to access elements in an array are implicitly coerced to strings? Well, sure ‚Äî they‚Äôre objects, and their indexes are their keys, and keys are strings (or Symbols). What happens if you use call() to give this a string literal value? There‚Äôs only one way to find out ‚Äî two ways to find out, if you factor in strict mode.

Either web components have come a long way since the last time I caught myself daydreaming about snacks at a conference, or I let my initial fear of them get in the way of truly knowing them ‚Äî probably both.

I‚Äôm here to tell you that you‚Äîyes, you‚Äîcan create a web component. Let‚Äôs leave our distractions, fears, and even our snacks at the door for a moment and do this together.

Let‚Äôs start with the <template>

A <template> is an HTML element that allows us to create, well, a template‚Äîthe HTML structure for the web component. A template doesn‚Äôt have to be a huge chunk of code. It can be as simple as:

<template>
  <p>The Zombies are coming!</p>
</template>

The <template> element is important because it holds things together. It‚Äôs like the foundation of building; it‚Äôs the base from which everything else is built. Let‚Äôs use this small bit of HTML as the template for an <apocalyptic-warning> web component‚Äîyou know, as a warning when the zombie apocalypse is upon us.

Then there‚Äôs the <slot>

<slot> is merely another HTML element just like <template>. But in this case, <slot> customizes what the <template> renders on the page.

<template>
  <p>The <slot>Zombies</slot> are coming!</p>
</template>

Here, we‚Äôve slotted (is that even a word?) the word ‚ÄúZombies‚Äù in the templated markup. If we don‚Äôt do anything with the slot, it defaults to the content between the tags. That would be ‚ÄúZombies‚Äù in this example.

Using <slot> is a lot like having a placeholder. We can use the placeholder as is, or define something else to go in there instead. We do that with the name attribute.

<template>
  <p>The <slot name="whats-coming">Zombies</slot> are coming!</p>
</template>

The name attribute tells the web component which content goes where in the template. Right now, we‚Äôve got a slot called whats-coming. We‚Äôre assuming zombies are coming first in the apocalypse, but the <slot> gives us some flexibility to slot something else in, like if it ends up being a robot, werewolf, or even a web component apocalypse.

Using the component

We‚Äôre technically done ‚Äúwriting‚Äù the component and can drop it in anywhere we want to use it.

<apocalyptic-warning>
  <span slot="whats-coming">Halitosis Laden Undead Minions</span>
</apocalyptic-warning>

<template>
  <p>The <slot name="whats-coming">Zombies</slot> are coming!</p>
</template>

See what we did there? We put the <apocalyptic-warning> component on the page just like any other <div> or whatever. But we also dropped a <span> in there that references the name attribute of our <slot>. And what‚Äôs between that <span> is what we want to swap in for ‚ÄúZombies‚Äù when the component renders.

Note

Here‚Äôs a little gotcha worth calling out: custom element names must have a hyphen in them. It‚Äôs just one of those things you‚Äôve gotta know going into things. The spec prescribes that to prevent conflicts in the event that HTML releases a new element with the same name.

Still with me so far? Not too scary, right? Well, minus the zombies. We still have a little work to do to make the <slot> swap possible, and that‚Äôs where we start to get into JavaScript.

Registering the component

As I said, you do need some JavaScript to make this all work, but it‚Äôs not the super complex, thousand-lined, in-depth code I always thought. Hopefully I can convince you as well.

You need a constructor function that registers the custom element. Otherwise, our component is like the undead: it‚Äôs there but not fully alive.

Here‚Äôs the constructor we‚Äôll use:

// Defines the custom element with our appropriate name, <apocalyptic-warning>
customElements.define("apocalyptic-warning",

  // Ensures that we have all the default properties and methods of a built in HTML element
  class extends HTMLElement {

    // Called anytime a new custom element is created
    constructor() {

      // Calls the parent constructor, i.e. the constructor for `HTMLElement`, so that everything is set up exactly as we would for creating a built in HTML element
      super();

      // Grabs the <template> and stores it in `warning`
      let warning = document.getElementById("warningtemplate");

      // Stores the contents of the template in `mywarning`
      let mywarning = warning.content;

      const shadowRoot = this.attachShadow({mode: "open"}).appendChild(mywarning.cloneNode(true));
    }
  });

I left detailed comments in there that explain things line by line. Except the last line:

const shadowRoot = this.attachShadow({mode: "open"}).appendChild(mywarning.cloneNode(true));

We‚Äôre doing a lot in here. First, we‚Äôre taking our custom element (this) and creating a clandestine operative‚ÄîI mean, shadow DOM. mode: open simply means that JavaScript from outside the :root can access and manipulate the elements within the shadow DOM, sort of like setting up back door access to the component.

From there, the shadow DOM has been created and we append a node to it. That node will be a deep copy of the template, including all elements and text of the template. With the template attached to the shadow DOM of the custom element, the <slot> and slot attribute take over for matching up content with where it should go.

Check this out. Now we can plop two instances of the same component, rendering different content simply by changing one element.

Styling the component

You may have noticed styling in that demo. As you might expect, we absolutely have the ability to style our component with CSS. In fact, we can include a <style> element right in the <template>.

<template id="warningtemplate">
  <style>
    p {
      background-color: pink;
      padding: 0.5em;
      border: 1px solid red;
    }
  </style>

    <p>The <slot name="whats-coming">Zombies</slot> are coming!</p>
</template>

This way, the styles are scoped directly to the component and nothing leaks out to other elements on the same page, thanks to the shadow DOM.

Now in my head, I assumed that a custom element was taking a copy of the template, inserting the content you‚Äôve added, and then injecting that into the page using the shadow DOM. While that‚Äôs what it looks like on the front end, that‚Äôs not how it actually works in the DOM. The content in a custom element stays where it is and the shadow DOM is sort of laid on top like an overlay.

And since the content is technically outside the template, any descendant selectors or classes we use in the template‚Äôs <style> element will have no affect on the slotted content. This doesn‚Äôt allow full encapsulation the way I had hoped or expected. But since a custom element is an element, we can use it as an element selector in any ol‚Äô CSS file, including the main stylesheet used on a page. And although the inserted material isn‚Äôt technically in the template, it is in the custom element and descendant selectors from the CSS will work.

apocalyptic-warning span {
  color: blue;
}
See The ______ Are Coming and They're Blue... by anon on CodePen.

But beware! Styles in the main CSS file cannot access elements in the <template> or shadow DOM.

Let‚Äôs put all of this together

Let‚Äôs look at an example, say a profile for a zombie dating service, like one you might need after the apocalypse. In order to style both the default content and any inserted content, we need both a <style> element in the <template> and styling in a CSS file.

The JavaScript code is exactly the same except now we‚Äôre working with a different component name, <zombie-profile>.

customElements.define("zombie-profile",
  class extends HTMLElement {
    constructor() {
      super();
      let profile = document.getElementById("zprofiletemplate");
      let myprofile = profile.content;
      const shadowRoot = this.attachShadow({mode: "open"}).appendChild(myprofile.cloneNode(true));
    }
  }
);

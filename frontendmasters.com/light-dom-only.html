When people talk about native Web Components, the Shadow DOM comes up a lot. I have extremely mixed feelings about the Shadow DOM. On one hand, it‚Äôs a powerful scoping tool. That‚Äôs what the Shadow DOM is. What it isn‚Äôt is required for ‚ÄúHTML abstraction‚Äù, which I feel like it gets confused for. It had me confused for a while, anyway. The Guts of a Custom Element Can Become Light DOM

Here‚Äôs a custom element defined as alert-machine:

  <button>I'm a button.</button>
</alert-machine>

In order for that to do anything, we need to execute some JavaScript that defines that element and its functionality:

  connectedCallback() {
    this.querySelector("button").addEventListener("click", () => {
      alert("Alert machine strikes again.");
    });
  }
}

customElements.define("alert-machine", AlertMachine);

I kinda prefer the slightly more elaborate setup with a constructor that allows for methods and such:

  constructor() {
    super();
  }

  connectedCallback() {
    this.querySelector("button").addEventListener("click", this.doAlert);

    alert("Alert machine strikes again.");
  }
}

  2. Well, that‚Äôs pretty limiting. I don‚Äôt want to have to be in charge of adding in all the needed HTML for every single component. That doesn‚Äôt bring very much value.

I don‚Äôt like styling the Shadow DOM

That‚Äôs kind of the rub here, for me. The main reason I‚Äôm so hot on Light DOM is that I find the styling story of Web Components using Shadow DOM annoying.

  • Styling very specific things with ::part is a very niche styling thing and to me, not a real styling solution.
  • Styling by documenting somewhere that certain CSS --custom-properties are in use is also limited and not a real styling solution.
  • Styling by injecting a <style> tag into some template literal in the JavaScript itself feels awkward and ad hoc to me and I‚Äôm not a fan.
  • Styling with an adopted stylesheet means an additional web request for each component or back to the template literal thing which is either awkward or slow.

I don‚Äôt dislike that these options exist, I just don‚Äôt‚Ä¶ like them. I‚Äôd rather be able to use the best styling API ever: regular CSS. There is some hope here, the idea of ‚Äúopen stylable shadow roots‚Äù (WICG/webcomponents) might take hold.

So what are we giving up again?

If you go all-Light-DOM with a Web Component, you lose:

  • Slots

You gain the ability to use regular ol‚Äô CSS from the parent page to style the thing. And, perhaps, a stronger reminder that Web Components should have good default HTML.
